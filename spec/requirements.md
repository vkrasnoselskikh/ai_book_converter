# Requirements Document: AI Book Converter CLI

## Introduction

Проект должен предоставлять CLI-утилиту для преобразования книг в форматах PDF и DJVU в электронную книгу,
пригодную для чтения на Amazon Kindle Paperwhite. Утилита должна отправлять исходный документ в "MistralOCR",
получать структурированный OCR-результат, обрабатывать контент постранично и собирать итоговую книгу в формате MOBI.

## Glossary

- **Source document** - исходная книга в формате PDF или DJVU.
- **OCR job** - единичный процесс распознавания и сборки книги для одного исходного документа.
- **Job directory** - временная директория с артефактами обработки конкретной книги.
- **Body block** - основной полезный контент страницы, предназначенный для переноса в итоговую книгу.
- **Header block** - верхний служебный блок страницы, который не должен попадать в итоговую книгу.
- **Footer block** - нижний служебный блок страницы, содержащий сноски или иные примечания страницы.
- **Endnotes** - раздел книги в конце итогового файла, в который переносятся сноски из footer-блоков.
- **Intermediate artifacts** - временные файлы обработки: JSON-ответы, извлеченные изображения, промежуточная разметка.

## Requirements

### 1. CLI and Input Handling

**ID:** book-converter.1

**User Story:** Как пользователь, я хочу запускать конвертацию книги из командной строки, чтобы преобразовать исходный файл в Kindle-совместимую книгу без ручной обработки.

#### Acceptance Criteria

1.1. КОГДА пользователь запускает CLI, система ДОЛЖНА принимать путь к исходному документу как обязательный аргумент.
1.2. КОГДА пользователь передает исходный документ, система ДОЛЖНА поддерживать форматы PDF и DJVU.
1.3. ЕСЛИ путь к исходному документу не существует, система ДОЛЖНА завершаться с понятной ошибкой.
1.4. ЕСЛИ расширение файла не поддерживается, система ДОЛЖНА завершаться с понятной ошибкой.
1.5. КОГДА пользователь запускает обработку, система ДОЛЖНА позволять указать рабочую директорию задания либо создать ее автоматически.
1.6. КОГДА пользователь запускает обработку, система ДОЛЖНА позволять указать модель OCR и параметры выходной сборки книги.

#### Functional Tests

- `tests/functional/test_cli_conversion.py` - запуск CLI с PDF и DJVU.
- `tests/functional/test_cli_validation.py` - ошибки на отсутствующем файле и неподдерживаемом формате.

### 2. OCR Job Lifecycle

**ID:** book-converter.2

**User Story:** Как пользователь, я хочу, чтобы конвертация выполнялась как воспроизводимый пайплайн с сохраняемым состоянием, чтобы можно было анализировать и при необходимости возобновлять обработку.

#### Acceptance Criteria

2.1. КОГДА начинается обработка книги, система ДОЛЖНА создать job directory во временной директории или в пользовательском пути.
2.2. КОГДА job directory создана, система ДОЛЖНА сохранять в ней состояние пайплайна.
2.3. КОГДА выполняется OCR job, система ДОЛЖНА сохранять сырой JSON-ответ "MistralOCR" в job directory.
2.4. КОГДА выполняется OCR job, система ДОЛЖНА сохранять промежуточные артефакты обработки в job directory.
2.5. ЕСЛИ обработка прерывается после создания job directory, система ДОЛЖНА сохранять достаточно данных для диагностики и возобновления.

#### Functional Tests

- `tests/functional/test_job_directory.py` - создание и повторное использование job directory.
- `tests/functional/test_resume_pipeline.py` - возобновление пайплайна из сохраненного состояния.

### 3. Mistral OCR Integration

**ID:** book-converter.3

**User Story:** Как пользователь, я хочу, чтобы документ отправлялся в "MistralOCR" и возвращался в структурированном виде, чтобы система могла собрать из него корректную электронную книгу.

#### Acceptance Criteria

3.1. КОГДА начинается OCR-этап, система ДОЛЖНА загрузить исходный документ в "MistralOCR".
3.2. КОГДА документ загружен, система ДОЛЖНА запросить обработку документа через OCR API.
3.3. КОГДА OCR API возвращает результат, система ДОЛЖНА получить постраничный структурированный JSON.
3.4. КОГДА OCR API возвращает результат, система ДОЛЖНА сохранять markdown-контент страницы, header, footer и сведения об изображениях.
3.5. ЕСЛИ OCR API возвращает ошибку, система ДОЛЖНА завершать текущий этап с диагностируемым сообщением и сохраненными промежуточными данными.

#### Functional Tests

- `tests/functional/test_mistral_fixture_pipeline.py` - обработка документа на основе сохраненного OCR fixture.
- `tests/functional/test_mistral_api_error.py` - реакция на ошибку OCR API.

### 4. Page Content Extraction

**ID:** book-converter.4

**User Story:** Как пользователь, я хочу, чтобы полезный контент книги корректно извлекался со страниц, чтобы итоговая книга содержала только читаемый текст и иллюстрации.

#### Acceptance Criteria

4.1. КОГДА обрабатывается страница OCR-результата, система ДОЛЖНА игнорировать header block.
4.2. КОГДА обрабатывается страница OCR-результата, система ДОЛЖНА включать body block в основной поток текста книги.
4.3. КОГДА body block содержит разметку markdown, система ДОЛЖНА сохранять ее семантику при преобразовании в итоговую книгу.
4.4. КОГДА на странице отсутствует body block, система ДОЛЖНА пропускать страницу без ошибки и фиксировать это в промежуточных данных.
4.5. ЕСЛИ OCR-ответ содержит неожиданные или пустые блоки, система ДОЛЖНА обработать их без аварийного завершения.

#### Functional Tests

- `tests/functional/test_page_content_selection.py` - исключение header и включение body.
- `tests/functional/test_empty_page_blocks.py` - обработка пустых и неполных страниц.

### 5. Footnotes and Endnotes

**ID:** book-converter.5

**User Story:** Как пользователь, я хочу, чтобы сноски из нижних колонтитулов не терялись и оставались читаемыми на Kindle, чтобы итоговая книга сохраняла смысл оригинала в reflowable-формате.

#### Acceptance Criteria

5.1. КОГДА OCR-результат страницы содержит footer block со сносками, система ДОЛЖНА извлекать их как примечания страницы.
5.2. КОГДА сноска извлечена, система ДОЛЖНА исключать footer block из основного потока текста страницы.
5.3. КОГДА итоговая книга собирается, система ДОЛЖНА переносить извлеченные сноски в отдельный раздел endnotes в конце книги.
5.4. КОГДА текст body содержит ссылки или маркеры сносок, система ДОЛЖНА формировать в итоговой книге навигационные ссылки между местом упоминания и соответствующей записью в endnotes.
5.5. ЕСЛИ footer block не удается надежно сопоставить с маркером в body, система ДОЛЖНА сохранить текст примечания в endnotes как несвязанную сноску, а не терять его.

#### Functional Tests

- `tests/functional/test_endnotes_generation.py` - перенос footer в конец книги.
- `tests/functional/test_endnotes_links.py` - генерация ссылок между body и endnotes.
- `tests/functional/test_unmatched_footnotes.py` - сохранение несвязанных примечаний.

### 6. Image Preservation

**ID:** book-converter.6

**User Story:** Как пользователь, я хочу, чтобы иллюстрации из книги сохранялись и попадали в итоговую книгу, чтобы электронная версия не теряла визуальный контент.

#### Acceptance Criteria

6.1. КОГДА OCR-ответ страницы содержит изображения, система ДОЛЖНА извлекать их из OCR-ответа и сохранять в job directory.
6.2. КОГДА изображения извлечены, система ДОЛЖНА включать их в итоговую книгу в местах, соответствующих body-контенту страницы.
6.3. ЕСЛИ OCR API возвращает изображения в base64-представлении, система ДОЛЖНА декодировать и сохранять их в файловой системе.
6.4. ЕСЛИ изображение повреждено или отсутствует, система ДОЛЖНА продолжать сборку книги, сохраняя диагностическую информацию.

#### Functional Tests

- `tests/functional/test_image_extraction.py` - сохранение изображений из OCR fixture.
- `tests/functional/test_image_embedding.py` - встраивание изображений в итоговую книгу.
- `tests/functional/test_broken_image_payload.py` - деградация при некорректном изображении.

### 7. Kindle-Oriented Book Assembly

**ID:** book-converter.7

**User Story:** Как пользователь Kindle Paperwhite, я хочу получить книгу в формате MOBI с удобочитаемой структурой, чтобы читать результат OCR как обычную электронную книгу.

#### Acceptance Criteria

7.1. КОГДА основной текст, изображения и endnotes подготовлены, система ДОЛЖНА собрать итоговую книгу в формате MOBI.
7.2. КОГДА итоговая книга собирается, система ДОЛЖНА использовать reflowable-разметку, пригодную для экранов Amazon Kindle Paperwhite.
7.3. КОГДА итоговая книга создается, система ДОЛЖНА сохранять логическую последовательность страниц исходного документа.
7.4. КОГДА итоговая книга создается, система ДОЛЖНА включать содержание или иную навигацию по крупным структурным блокам, если они доступны из OCR-данных.
7.5. КОГДА итоговая книга создана, система ДОЛЖНА сохранять ее в явно определенный выходной путь.

#### Functional Tests

- `tests/functional/test_mobi_output.py` - генерация MOBI-файла.
- `tests/functional/test_kindle_navigation.py` - наличие навигации и endnotes в итоговой книге.

### 8. Test Data and Mocking Strategy

**ID:** book-converter.8

**User Story:** Как разработчик, я хочу использовать один реальный OCR-ответ в качестве тестового fixture, чтобы последующие тесты были детерминированными и не зависели от внешнего API.

#### Acceptance Criteria

8.1. КОГДА подготавливаются тестовые данные, система ДОЛЖНА использовать книгу из директории `tests/assets`.
8.2. КОГДА создается OCR fixture для тестов, система ДОЛЖНА отправлять тестовую книгу в "MistralOCR" только один раз вручную или отдельной подготовительной командой.
8.3. КОГДА OCR fixture сохранен, автоматические тесты ДОЛЖНЫ использовать сохраненный JSON вместо повторного сетевого вызова.
8.4. КОГДА выполняются unit и functional tests, они НЕ ДОЛЖНЫ требовать доступа к реальному OCR API по умолчанию.
8.5. ЕСЛИ OCR fixture отсутствует, система ДОЛЖНА выдавать понятную инструкцию по его подготовке вместо молчаливого падения.

#### Functional Tests

- `tests/functional/test_fixture_loading.py` - использование сохраненного OCR JSON.
- `tests/functional/test_no_live_api_in_default_tests.py` - подтверждение отсутствия сетевого вызова по умолчанию.
